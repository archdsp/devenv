# 전역 작업 규칙 (모든 프로젝트 공통)

이 파일은 user-level 메모리이며, 작업 디렉터리와 무관하게 모든 세션에 적용된다.

## 1. Python / Anaconda 환경

- Python 프로젝트에서는 **항상 Anaconda(conda) 환경**을 사용한다. 시스템 Python이나 venv보다 conda 환경을 우선한다.
- 작업 시작 시 적절한 conda 환경을 확인/활성화한다.
  - 프로젝트에 `environment.yml`이 있으면 해당 환경을 우선 사용한다.
  - 활성화할 환경이 명확하지 않으면 `conda env list`로 후보를 보여주고 **어떤 환경을 쓸지 사용자에게 물어본다.** 임의로 base를 쓰지 않는다.
- Python 실행/패키지 설치 명령은 활성화된 conda 환경 안에서 수행한다 (`conda activate <env>` 후 실행, 또는 `conda run -n <env> ...`).

## 2. 작업 진행 기록 — WIP.md

- 프로젝트 루트의 `WIP.md`에 작업 진행 상황을 **수시로 기록**한다. 특히 작업이 중단되거나 마무리될 때, 다음에 이어서 할 수 있도록 아래 내용을 남긴다.
  - **현재 목표**: 지금 무엇을 하려는지
  - **완료한 것**: 끝난 항목
  - **진행 중 / 중단 지점**: 어디까지 했고 무엇이 막혀 있는지
  - **다음 할 일**: 이어서 해야 할 구체적 단계
  - **참고**: 관련 파일 경로, 명령어, 결정 사항
- `WIP.md`는 한글 마크다운으로 작성하고, 매번 덮어쓰기보다 현재 상태를 정확히 반영하도록 갱신한다.

## 3. 브랜치 정책

- **작업은 항상 `dev` 브랜치에서** 진행한다.
  - 세션 시작 시 현재 브랜치를 확인한다. `dev`가 아니면 `dev`로 전환한다.
  - `dev` 브랜치가 **없으면 자동으로 만들지 말고**, 상황을 알린 뒤 **사용자에게 어떻게 할지 물어본다.**
- **WIP 커밋은 `wip` 브랜치 하나**에 모은다 (`dev`를 더럽히지 않는다).
  - 토큰/컨텍스트가 끊길 것 같거나(자동 압축 직전 등), 작업이 길어져 중간 저장이 필요한 시점에:
    1. `WIP.md`를 최신 상태로 갱신하고
    2. `wip` 브랜치로 전환(없으면 생성)한 뒤
    3. 현재까지의 작업을 `wip` 브랜치에 커밋한다.
  - 참고: 컨텍스트 소진 시점을 정확히 예측하긴 어려우므로, **자연스러운 체크포인트마다 best-effort로** WIP 커밋을 남긴다.

## 4. 커밋 메시지 규칙

- `~/.gitmessage` 템플릿이 존재하면 **그 양식(섹션 구조)을 따른다.** (현재 `git config commit.template`에 연결되어 있음)
  - 제목: `Type : 한 줄 요약` (Type 후보: Fix / Hack / Refactor / Style / Docs / Test / Chore)
  - 본문 섹션: `Description`, `Fix`(+/-), `Refactor`(+/-), `How to test`, `Reference(선택)`
- 커밋 메시지는 **한글 마크다운으로 길고 자세하게** 작성한다.
  - 무엇을 왜 바꿨는지(Description)
  - **테스트 완료 여부**와 **구체적인 작동/실행 방법**(How to test)을 반드시 포함한다.
  - 관련 이슈/참고 링크가 있으면 Reference에 적는다.
- 커밋 메시지 마지막에 co-author 라인을 포함한다:
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`
- `git commit -m`으로 여러 줄/마크다운을 넣을 때는 `-m`을 여러 개 쓰거나 heredoc(`-F`)을 사용해 줄바꿈과 서식이 유지되게 한다.

## 5. 커밋 워크플로 — lint → 리뷰 → atomic 커밋 → PR

- 커밋을 만들기 전에 **항상** 아래 순서를 따른다. (`/commit` 슬래시 명령으로 한 번에 실행할 수 있다.)
  1. **Lint/Format**: 프로젝트에 맞는 린터·포매터를 변경 파일에 돌리고, 에러가 없을 때까지 고친다. (Python은 시스템 python이 아니라 **활성 conda 환경**에서 실행한다.)
  2. **Ultracode 리뷰**: 변경 diff를 깊게(ultrathink) 리뷰한다. 가능하면 전용 리뷰 서브에이전트에 맡겨 정확성·보안·단순화 관점에서 점검하고, **치명적 문제는 커밋 전에 고친다.**
  3. **Atomic 커밋**: 변경을 논리적으로 독립된 단위로 나눠 부분 스테이징(`git add -p`)하고, 4번 규칙의 메시지 양식으로 각각 커밋한다. 한 커밋에 여러 목적을 섞지 않는다.
  4. **PR**: 작업 브랜치를 push 한 뒤 PR을 생성한다. 단, 사용자가 원치 않으면 PR은 만들지 않으며, 보호 브랜치(main 등)에는 직접 push 하지 않는다.
- 자세한 단계는 `~/.claude/commands/commit.md`(`/commit`)를 참고한다.

## 6. 일반

- 위 규칙은 기본 동작이며, 사용자가 특정 작업에서 다르게 지시하면 그 지시를 우선한다.
